# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A **reusable Telegram ↔ LLM bot engine** on **Cloudflare Workers** (strict TypeScript ES-modules under `src/`): it bridges Telegram webhooks to the OpenRouter LLM API and layers on per-chat config, long-term memory (RAG), summaries, vision, dedup and a daily cron. The bot's **personality** — its name, voice and "fun" commands — is a **swappable persona pack** loaded at runtime, and is **not in this repo**: it's supplied at build time via the `PERSONA_PACK` env-path (see **Persona pack** below). Without a pack the engine runs neutral (no personality). Engine code, comments and engine-level strings are in **English**; user-facing UI strings are localized via `t()` from `src/i18n/*.json` (default **English** — `DEFAULT_LANG = "en"`, env `BOT_LANG` default `en`, per-chat `/lang` or `/config lang`) — see **Localization** below. Match that style when editing (no hardcoded user-facing literals — add a key to the locale files).

This is a **wrangler project**, but the engine is a **library — it does not deploy itself**: the real `wrangler.jsonc` (worker name, live resource ids, secrets, migrations-apply) lives in the **consuming deployment project**, which pulls this engine and injects a pack via `PERSONA_PACK`. The `wrangler.jsonc` here is a generic template (placeholder ids) for `wrangler types` / a fork's local dev. The worker source is **typed TypeScript ES-modules under `src/`**, layered bottom-up (each imports only from lower layers; no cycles):

`types.ts` (shared types) · `constants.ts` · `i18n/` (locale JSON + the `t()` resolver) → `utils.ts` (pure helpers incl. `makeCtx`/`shouldAnswer`/visual + time helpers) / `prompts.ts` / `rag.ts` (Vectorize RAG: embed/upsert/query/delete helpers) / `persona/registry.ts` (persona contract + active-pack singleton) → `storage.ts` (D1: per-chat state row + message rows) → `telegram.ts` · `llm.ts` · `config.ts` → `vision.ts` · `summary.ts` (incremental `/summary` core) · `curation.ts` (fact extraction for RAG) → `commands.ts` (engine command plugins + `COMMANDS` map) → `flow.ts` (routing + `handleTelegramMessage`) → **`index.ts`** (entry: `export default { fetch, scheduled }` + a `export *` barrel re-exporting every engine module for the tests). `persona/registry.ts` is the lowest persona layer (imports only types/constants); the active pack registers via a side-effect `import "./persona/active"` placed **first** in `index.ts`. `wrangler.jsonc` `main` = `src/index.ts`; wrangler/esbuild bundles the TS entry. Imports are **extensionless** (`./constants`) under `moduleResolution: "Bundler"`.

Bindings (declared in `wrangler.jsonc`): `KV` (general-purpose KV, keys namespaced by prefix — `dedup:*`, etc.), `DB` (D1 — per-chat state + message rows + `memories`), `AI` (Workers AI, RAG embeddings) and `VECTORIZE` (semantic memory index). **Secrets** (`TELEGRAM_BOT_TOKEN`, `OPENROUTER_API_KEY`, `OPENROUTER_PROVISIONING_KEY`) are write-only on the worker (set by the deployment project). `WORK.md` (the live to-do, driven by the `tracking-work` skill) is kept **outside** this repo — it is shared between the engine and the pack. `.claude/skills/` holds the project skills.

## Running, testing & deploying

A Cloudflare Worker driven by a Telegram webhook (`POST` of an Update → `fetch`). Persistence: **D1** (`DB` binding — per-chat state + message rows) for chat data, **KV** (`KV` binding, general-purpose with prefixed keys) for `dedup:<id>` flags (5-min TTL). D1 is the sole source of truth.

**Tests:** `npm test` runs the offline **Vitest** suite (~315 neutral / ~364 with a pack; **real D1 via a `node:sqlite` shim** in the harness, mocked `fetch`) — split across `*.test.mjs` under `.claude/skills/testing-worker/` (shared `harness.mjs`), run in parallel. Seed/read D1 in tests with `seedChat`/`dbChat`/`dbHistory`. `npm run typecheck` = `npm run check` = `tsc --noEmit` (strict). The suite reaches worker internals through the **`export *` barrel** in `src/index.ts` — **inert at runtime** (Cloudflare uses only `export default`). To unit-test a new internal function, just `export` it from its module — the barrel re-exports it automatically. Persona-specific tests live with the pack in the **private** repo (`*.persona.test.mjs`) and are staged next to the engine tests by CI / locally; the committed public suite is persona-free. See the **`testing-worker`** skill.

**CI (the engine does not deploy).** A push to **`master`** triggers GitHub Actions (`.github/workflows/ci.yml`): `npm ci` → typecheck + tests on the **neutral** persona (no `PERSONA_PACK` → `src/persona/active.ts` = `import "./default"`). That's it — no deploy. The normal path is **commit + push** (Claude is authorized to push — `Bash(git push:*)` allow-listed). **Deployment lives in the consuming project**, which pulls this engine, injects its pack via `PERSONA_PACK`, applies D1 migrations (before deploy — `wrangler deploy` alone does *not* apply them, and new code on a missing column makes `flushChatData` throw on every flush, silently swallowed → all chat state stops persisting), and runs `wrangler deploy` against its own resources/secrets. The **`tracking-work`** skill encodes the engine procedure (record in `WORK.md` → implement → typecheck+test gate → **commit → push** → CI verifies → prune).

### Environment variables (read in `getGlobalConfig`)

Engine behavior knobs, all optional with defaults: `OPENROUTER_HOST` (`https://openrouter.ai/api/v1`), `OPENROUTER_MODEL` (`openrouter/free`), `OPENROUTER_VISION_MODEL`, `OPENROUTER_SUMMARY_MODEL`, `OPENROUTER_TITLE`, `BOT_USERNAME` (`bot`), `BOT_NAME` (`Bot`), `BOT_LANG` (UI language, `en`), `BOT_TZ` (IANA timezone for history timestamps + the daily-summary cron gate, `UTC`), `MAX_HISTORY_CHARS` (8000), `MAX_TOKENS` (4000, response-length cap → `cfg.maxTokens`; 0 = omit, model default), `ANSWER_PROB` (0.1), `ENABLE_VISION` (off), `ENABLE_REASONING` (on), `VISION_DETAIL` (`low`), `VISION_HD_WORDS`, `ENABLE_RAG` (off — long-term memory), `RAG_TOP_K` (5), `RAG_MIN_SCORE` (0.5), `LLM_LOG`. Command names are fixed (no env-based renaming). `ADMIN_USERNAMES` is a CSV list of admin usernames (no `@`, case-insensitive) → `cfg.adminUsernames`; unset → empty (no admins). `ADMIN_CHAT_IDS` (optional) is a CSV list of chat ids (negatives = groups OK) that receive `reportError` critical alerts — unset → log-only. `botId` is derived from the token. **Persona-specific env vars** (the pack's switches/probabilities) are documented in the extensions repo and read by the pack via a cast — the engine `Env` type does not name them. `daily_summary` is a per-chat `/config` key (default off), not an env var.

## Architecture

### Request lifecycle (`fetch` → `handleTelegramMessage`)

- **Always returns HTTP 200 `"ok"`**, even on internal error — a non-200 makes Telegram redeliver the same Update (duplicate replies). All handler errors are caught and swallowed. Because that hides failures, the silent-but-catastrophic sites route through **`reportError(env, where, err, {critical})`** (`telegram.ts`): always a structured `console.error` (for `wrangler tail`/logpush), plus — when `critical` and `ADMIN_CHAT_IDS` is set — a Telegram alert to each admin chat (CSV list), throttled via KV (`errnotify:<where>`, 10-min TTL). Critical sites: `flushChatData` failure (persistence silently stops), `getChatData` `_loadFailed`, and the top-level `scheduled`/`handleTelegramMessage` crashes. `reportError` lives in `telegram.ts` and is called from `flow.ts`/`index.ts` (above it) — never from `storage.ts` (would cycle).
- **Processing is synchronous (`await`), deliberately NOT `ctx.waitUntil`.** `waitUntil` background work can be killed before the LLM reply reaches Telegram. The worker must stay alive until the reply is sent; the LLM timeout (`LLM_TIMEOUT_MS`, 50s) is kept under Telegram's ~60s webhook wait.
- **Dedup:** each `update_id` writes a `dedup:<id>` KV flag with 5-min TTL; a repeat Update is ignored.

### State & persistence (D1)

Per-chat state is a row in the **`chats`** table (`role`, `paused`, `config`/`photo_cache`/`persona_state` as JSON, `spend`/`spend_count`, `name`, `summary`); **history is one row per message** in **`messages`** (`role`, `content`, `meta` JSON, `message_id`). `persona_state` is a generic JSON slot the persona owns (see **Persona pack** → personaState) — the engine never reads its keys. `ctx.chatData` (shape `DEFAULT_CHAT_DATA`) is the in-memory working copy. Types split the concern: **`ChatState`** = exactly the persisted `chats` columns; **`ChatData extends ChatState`** adds the recent history **window** + transient flags (`_dirty`, `_loadFailed`, `_preview`) that are never persisted. (The `_`-prefix on some `ChatState` fields is historical, not a "runtime" marker.) History is write-through (per-message INSERT), eliminating the read-modify-write race that dropped messages on concurrent updates.

Convention: **state — load once (`getChatData` → `chats` row + recent `messages` window), mutate `ctx.chatData` via mutators that set `_dirty`, flush once at the end (`flushChatData` UPSERTs the `chats` row) in a `finally`.** Mutators: `setRole`/`setPaused`/`saveChatConfig`/`addSpend`/`cachePhotoDesc`/`updatePersonaState` (shallow-merges a patch into `persona_state`). **History — write-through, immediate:** `appendHistory`/`updateHistoryMessage`/`clearChatHistory`/`dedupeChatHistory` are **async** and hit D1 directly (`INSERT OR IGNORE` + window-function trim `DELETE`); `await` them. `dedupeHistory`/`collapseConsecutiveDuplicates`/`trimHistoryByChars` are still the in-memory window helpers reused by `appendHistory`/`getChatData`.

### Config layering

`getGlobalConfig(env)` (env + hardcoded defaults + the active pack's `defaults(env)`) → `mergeConfig` overlays per-chat overrides from `chatData.config` (only keys present in `CONFIG_SCHEMA`) → `effectiveCfg` on `ctx.cfg`. `CONFIG_SCHEMA` is the source of truth for which keys are user-settable and how they validate (bool/float/int/string), composed as **engine schema ∪ persona schema**. Adding an **engine** `/config` option = add an `ENGINE_CONFIG_SCHEMA` entry (and usually an `ENGINE_CONFIG_GROUPS` entry for the help text); **persona** options come from the pack's `ConfigContribution`. Per-chat `model`/`vision_model`/`summary_model`/`max_tokens` overrides map onto `cfg.openrouterModel`/`cfg.visionModel`/`cfg.summaryModel`/`cfg.maxTokens`.

### Localization & timezone

UI strings are externalized to `src/i18n/*.json` and resolved through `t(lang, key, ...args)` (`src/i18n/index.ts`): a value is a string or an array of lines (joined with `\n`), `{0}`/`{1}` are positional args, a missing key falls back to `DEFAULT_LANG` (`"en"`) then to the key itself. `tList(lang, key)` returns the raw array (no fallback) for input-matching vocabularies; `tRaw(lang, key)` returns a single string or `undefined` (lang→`en` fallback; arrays joined). `cfg.lang` (env `BOT_LANG`, default `en`; per-chat `/lang <code>` or `/config lang <code>`) selects the language; every engine builder takes `cfg.lang`. **Locales are folder-discovered — the code hardcodes no language list.** `scripts/select-persona.mjs` scans `src/i18n/*.json` (engine) + the staged pack's `src/persona/_pack/i18n/*.json` (persona) and writes the **gitignored** `src/i18n/_generated.ts` static import manifest (Workers have no runtime fs, so locales are baked in at generate time); `MESSAGES` = engine ∪ persona tables merged per locale, and `LOCALES = Object.keys(MESSAGES)`. So **adding a language = drop a `src/i18n/<code>.json` file** — `<code>` is then usable via `/lang`/`/config lang` with no code change and no registration. The **persona** localizes its own texts in its **own `i18n/<lang>.json` folder** (merged in by the same generate step): the `PersonaTexts` fields under `persona_*` keys (resolved by `getPersonaTexts(lang)` via `tRaw`), and `ConfigMeta.desc` / `CONFIG_GROUPS` keys are locale **keys** (a persona may also pass a literal — `t()` returns unknown keys as-is). **Every** localized string a pack produces lives in its `i18n/<lang>.json` and resolves via `t()`/`tList()`: displayed command/status output, prompt-builder instruction lines, prompt seeds / synthetic user-turns, the injected flavor/state lines, **and** the quick-reply responses. The **only** things that stay inline in a pack are (a) the non-localized identity (`wakeWords`/`usernameAliases`) and (b) input-matching triggers — the quick-reply `test` regexes and the `tokenTable` **keys** — which match incoming messages rather than being output. The engine's **neutral/personaless fallbacks are themselves engine i18n keys** (`neutral_fallback_error`/`neutral_fallback_no_credits`/`neutral_target_name`/`neutral_info_title`), so a no-pack deployment localizes per `cfg.lang`; likewise `/config` booleans render via `cfg_on`/`cfg_off`, the `reportError` admin alert via `err_admin_alert`, and `getUserName(msg, lang)` falls back to `name_user_fallback`.

`cfg.timezone` (env `BOT_TZ`, default `UTC`, **deployment-wide** — not a `/config` key) drives the history-item timestamps (`tzStamp`, threaded through `toLLMMessages`) and the daily-summary cron gate (08:00 in that TZ). The Intl formatter is cached per TZ (`utils.ts`), with a UTC fallback for an invalid id.

### Message routing (in order)

1. **Edits** (`edited_message`) → `updateHistoryMessage` patches the stored item by `message_id`, bot does not re-reply.
2. **Visual** (photo/sticker, own or from a reply) → `handlePhotoMessage`, *unless* the message is a command (commands win).
3. Log incoming message to history.
4. If `paused`, only commands are processed.
5. **Quick replies** (`tryQuickReply`, from the pack) → **commands** (`tryCommand`) → **chat** (`handleChatMessage`).

`shouldAnswer` decides unprompted replies: always in private chats; in groups only on mention (`messageMentionsBot` = contains `@<botUsername>` or one of the pack's **wake words**) or random roll against `answer_prob`.

### Commands

Commands are **plugins** — the engine's core commands use the same `RegisteredCommand` contract as the pack. The engine registers them via `setEngineCommands(ENGINE_COMMAND_PLUGINS)` (in `commands.ts`), and `getAllCommands()` = engine ∪ persona is the single source that drives `COMMANDS`/`TECH_COMMANDS`/`LLM_COMMANDS`/`buildCommandRegex`. `COMMANDS` is a map of `type → async (ctx, mode) => string|null` (returning `null`/empty = stay silent). `buildCommandRegex` compiles a cached, ordered regex list from `getAllCommands()` (command name = each plugin's fixed `defaultCmd`; engine first, then the pack). Two important sets:
- `TECH_COMMANDS` — their output is **not** written to history (status/help/config replies), so the model doesn't echo them as its own speech. Includes the pack's `skipHistory` commands.
- `LLM_COMMANDS` — LLM-backed (shown with a "typing" indicator): the engine `summary` plus every persona command flagged `llm`. Content-generating ones are **stored** in history; `summary` is the exception — it's *also* in `TECH_COMMANDS`, so its output is **not** stored.

**Engine commands:** `help`, `config`, `info`, `model`, `memory`, `rp`, `stop`, `resume`, `summary`, `lang`, `alias`, and the hidden `admin`. Everything else (jokes, dice, mood, etc.) is **persona content** supplied by the pack.

`/help` is **additive**: `buildHelp(lang)` renders the engine's base command list (the `help_engine` locale key) and **appends** the active persona's `helpText` (`persona_helpText`) below it — the pack *adds* to the base list, it does not replace it (without a pack only the base shows). `/lang` (no arg) shows the current language + the discovered `LOCALES`; `/lang <code>` switches, or **errors (language unchanged)** if `<code>` is not a discovered locale; `/config lang <code>` applies the **same** validation (`setConfigParam` rejects an unknown code via `LOCALES.includes` — no silent fallback).

`/alias` manages **per-chat** display-name aliases (`username → name`): `/alias @user Name` sets one, bare `/alias` lists them, `/alias del @user` removes one (i18n keys `alias_*`). They live in the **reserved `aliases` object** on the per-chat config (`chats.config.aliases`, typed as `ChatConfig.aliases?: Record<string,string>` in `types.ts`) — **not** a `CONFIG_SCHEMA` scalar, so `mergeConfig` (which copies only `CONFIG_SCHEMA` keys) never leaks it into the effective `BotConfig`, and there's **no DB migration** (it rides the existing `config` JSON column). The name resolvers read it straight off `chatData.config` via the `chatAliases(ctx)` helper (`utils.ts`, returns `ctx.chatData.config.aliases ?? {}`) and merge it **over** the pack-static `usernameAliases` (per-chat wins). `resolveUserName`/`getUserName`/`pickTargetName`/`getUserMeta`/`buildUserItem` all take an optional `chatAliases`, and ctx-having call sites (`flow.ts` logging, `vision.ts`, the pack command/throw handlers) pass `chatAliases(ctx)` — so a `/alias` ALSO fixes the `[from:Name]` history tag the model sees (previously the tag used the raw `first_name`).

`/summary` builds an **incremental** LLM digest of *new* messages since the last command-summary boundary (`_cmdUptoId`, by `messages.id`; reset daily via `_cmdDay` in `cfg.timezone`), feeding the previous digest (`_summary`) as "already known — don't repeat" context. Refuses if too few new messages; `/memory forget all` (or `… history`) resets the boundaries + `_summary`, and `… all`/`… state` reset `persona_state` (to `getPersonaStateDefaults()`). Uses a separate `summary_model` if configured (`/model summary <id>`, env `OPENROUTER_SUMMARY_MODEL`) else the main model (`runLLMWithHistory` takes a `modelOverride`).

`/admin` is a hidden command gated to the **`ADMIN_USERNAMES`** list (`cfg.adminUsernames` — CSV env, case-insensitive; unset → empty list = no admins, the engine hardcodes no personal username) in private chats only; for anyone else it returns `null` (no hint it exists). It can inspect all chat sessions (from D1) and run **any** command in *other* chats via `chat_cmd` (output goes to the admin, not the target). LLM commands (`summary` + the pack's `llm` commands) run as a **preview**: their effects are NOT persisted to the target chat. The per-chat `chats`-row flush is skipped for `LLM_COMMANDS` (so `spend`/summary boundaries don't persist); additionally `targetCtx._preview` is set, which short-circuits `runMemoryCuration` — `summary`'s first step — so its **write-through** `addMemory` (which bypasses the flush) never writes curated facts into the target's memory. Control commands (`rp`/`config`/`memory`/…) do persist. `admin` itself is rejected (no recursion).

### LLM calls (`callOpenRouter`)

Always **streaming SSE**. Two abort timers: a hard cap (`LLM_TIMEOUT_MS`) and an idle timer (`LLM_IDLE_TIMEOUT_MS`) re-armed on every chunk — so a slow reasoning model isn't cut off as long as tokens keep flowing. `reasoning` config toggles `{exclude:true}` vs `{enabled:false}`. Cost from the final `usage` chunk is added via `addSpend`. Errors degrade to the localized fallback strings from `getPersonaTexts(cfg.lang)` — `fb.fallbackError`; HTTP 402 → `fb.fallbackNoCredits` (the pack's `persona_*` value, or the engine's `neutral_fallback_*` key without a pack). Fallback strings are **not** stored in history (`isFallbackMessage`, which reads them across all locales — both `persona_*` and `neutral_*`).

`runLLMWithHistory` → `toLLMMessages` builds `[system, ...history, user]`. History items carry `meta` and are rendered with a `formatWithMeta` prefix `[from:Name; <timestamp in cfg.timezone>]` (the timestamp comes from `meta.ts`, injected from the `messages.created_at` column on load — `tzStamp`; **not** the Telegram `message_id`, so the model cites date/time, not internal ids). The system prompt tells the model to ignore the tag and reference events by date/time; outgoing text is stripped of any `[from:...]` prefix in `sendAndStore`.

### Prompts (`assemblePrompt`)

Every prompt = localized instruction lines (`t(cfg.lang, …)`) + persona voice (custom `role` or the pack's `defaultVoice`) + the pack's extra lines from the optional `buildPromptLines(ctx)` hook. The engine knows **no** persona-state semantics: any per-chat flavor (e.g. a "mood" level) lives in `persona_state`, and the pack's `buildPromptLines` reads it and returns the lines to inject. The engine ships **default/reply/vision** builders (`buildDefaultPrompt`/`buildReplyPrompt`/`buildVisionPrompt`); the pack contributes its own content-command prompt builders. `assemblePrompt(instructions, ctx, memories)` reads the active pack per call, so with the neutral pack it emits a clean prompt with no persona lines.

### Vision

`handlePhotoMessage` and `describeCtxPhoto` handle photos and stickers (static sticker = the `.webp`; animated/video sticker = its thumbnail first frame). **Descriptions are cached by `file_unique_id`** (`photoCache`, capped at `PHOTO_CACHE_CAP`) — a re-seen image is answered from cache via a text-only call, never re-downloaded or re-sent to the vision model. Vision detail (`low`/`high`, a cost/quality tradeoff) is bumped to `high` when the caption contains a `visionHdWords` trigger. A separate `vision_model` can be configured. Album/media-group context is reconstructed from history `meta.media_group_id` + cached frame descriptions.

### Long-term memory = CURATED FACTS (`rag.ts` + `memories`)

Long-term memory stores **curated facts** (not every raw message): facts auto-extracted before each summary + added manually via `/memory add`. Source of truth = the **`memories`** D1 table (`id`,`chat_id`,`text`,`source`=`auto`|`manual`,`created_at`, migration 0003); the vector layer over **Cloudflare Vectorize** (`VECTORIZE`, a 1024-dim cosine index) + **Workers AI** embeddings (`AI`, `@cf/baai/bge-m3`) is for semantic recall. Everything is **best-effort** (`try/catch` + `withTimeout`), so a failure never breaks history persistence or the reply.

- **Fact vector**: id `m<chatId>:<memories.id>`, namespace `mem:<chatId>` (per-chat isolation), `metadata.text` carries the fact (capped `RAG_META_TEXT_CAP`) so recall needs no D1 round-trip.
- **Gating split**: `ragUpsertMemory`/`ragDeleteIds` gate on **bindings only** (NOT `cfg.rag`) — so `/memory add` embeds a fact even with RAG off, ready to recall later without backfill. **Recall** (`ragRetrieveMemories`) and **auto-curation** gate on `cfg.rag` (default off). When `ctx._preview` is set (admin preview of another chat) curation is skipped entirely so nothing is written through to the target.
- **Write** (`storage.addMemory`): INSERT `memories` row → `ragUpsertMemory`. `appendHistory` does **not** embed messages; raw messages live only in `messages`.
- **Auto-curation** (`curation.runMemoryCuration` + `parseExtractedFacts`, called **before every summary** — both the daily cron `runDailySummaries` and the `/summary` command — for chats with `rag` on, best-effort; NOT per chat reply): summarize the `messagesSince(_memUptoId)` delta (gated `MEM_CURATION_MIN_NEW`), extract facts from **user** messages only via a **neutral** plain prompt (`buildMemoryExtractionPrompt`, NOT `assemblePrompt` — no persona), `modelOverride=summaryModel`; `parseExtractedFacts` strips bullets/headings/refusals + caps + dedups; `addMemory("auto")` each; advance `_memUptoId` on LLM success (even 0 facts), not on fallback. The cron processes any chat with `rag` **or** `daily_summary` on.
- **Recall+inject** (`handleChatMessage`, default handler only): embed user text → `query` top-`rag_top_k` (clamped to 50) in `mem:<chatId>` → keep `score >= rag_min_score`, require `metadata.text` → inject **plain fact text** as a labeled **system-prompt** section (`assemblePrompt(..., memories)`), never into the history array.
- **Cleanup**: `/memory forget <target>` wipes a chosen slice — `all` / `history` / `memory` / `role` / `state` / `cache` / `spend` (a bare `/memory forget` returns usage help and wipes nothing). `all`/`history` → `clearChatHistory` + reset the summary/curation boundaries; `all`/`memory` → `clearMemories` (DELETE rows + `deleteChatMemoryVectors`) + reset `_memUptoId`. `/memory del <N>` → `deleteMemory` (one fact by its `/memory list` index). `appendHistory` protects un-curated rows (`id > _memUptoId`) — and un-summarized rows — from trimming when `rag`/`daily_summary` is on, bounded by `SUMMARY_BACKLOG_CAP`.
- Config: `rag` (bool, off; env `ENABLE_RAG`), `rag_top_k` (int 1–20, 5), `rag_min_score` (float 0–1, 0.5). `RAG_EMBED_MODEL` hardcoded (swap → re-index). Embed cost **not** in `addSpend`. Tests mock both bindings in `harness.mjs` (`makeAI`/`makeVectorize`/`seedMemories`/`dbMemories`).

### History maintenance

`appendHistory` runs `dedupeHistory` (by `role:message_id`) + `collapseConsecutiveDuplicates` + `trimHistoryByChars` (newest-first, char budget `history_chars`, plus `HISTORY_HARD_CAP_ITEMS` item cap). History is the bot's only *short-term* memory; semantic recall of older messages is the **RAG** layer above.

### Telegram output (`sendTelegramMessage`)

Outgoing text is treated as a lightweight Markdown that `toMarkdownV2` converts to valid Telegram MarkdownV2 — it **escapes everything except intentional paired markup** (`*bold*`, `_italic_`, `~strike~`, `||spoiler||`, `` `code` ``, ```` ```block``` ````, `[text](url)`). `**` is normalized to `*` and `##`–`####` headings to bold first. Sends try MarkdownV2, then fall back to plain text. Messages over `TELEGRAM_MSG_LIMIT` (4096) are split (surrogate-pair safe).

## Persona pack (the swappable personality)

The engine hardcodes **no** persona content. The personality is a **pack** the engine reads at runtime via the registry singleton (`src/persona/registry.ts`): `setPersona`/`getPersona` + `getPersonaCommands`/`getPersonaQuickReplies`/`getPersonaThrows`/`getPersonaConfig`/`getPersonaTexts(lang)`, plus `getAllCommands`/`getPersonaStateDefaults`. The `PersonaPack` contract — **all fields optional** (the neutral pack is `{}`):

- **`wakeWords`** (`string[]?`) and **`usernameAliases`** (`Record<string,string>?`) — the pack's **only** non-localized identity on the object: words that wake the bot in groups (substring) + pack-static `username (lowercase, no @) → display name` defaults. (Per-chat `/alias` entries — `chats.config.aliases` — merge **over** these in the name resolvers; the pack value is just the default. See **Commands** → `/alias`.)
- **`commands`** (`RegisteredCommand[]?`) — `{type, defaultCmd, handler, llm?, skipHistory?, state?}`; merged into `COMMANDS`/`buildCommandRegex`/`LLM`/`TECH`. `state` is this command's slice of the persona-state schema (see below).
- **`quickReplies`** (`QuickReplyRule[]?`) — substring/token-table reactions, gated by `cfgFlag`×`probKey`; iterated in order by `tryQuickReply`. The `responses` field is an i18n **key** whose value is the candidate array, and `tokenTable` maps an input token → an i18n key; `tryQuickReply` resolves them per `cfg.lang` via `tList` → `pickOne`. The `test` regexes and `tokenTable` **keys** match incoming messages and stay inline (input matching, not output).
- **`randomThrows`** (`RandomThrowKind[]?`) — weighted random content "throws"; `pickRandomRandomKind` + `RANDOM_HANDLERS`.
- **`config`** (`ConfigContribution?`) — `{schema, groups, presets, defaults(env)}`; merged into `CONFIG_SCHEMA`/`CONFIG_GROUPS`/`CONFIG_PRESETS` and `getGlobalConfig`.
- **`buildPromptLines(ctx)` / `infoLines(ctx)` / `adminFlags(state)`** (optional hooks) — extra lines the pack injects into every system prompt / the `/info` status / the `/admin` per-chat flag, derived from its own `persona_state`. The engine calls them blind to their meaning.

**There is no `texts`/`localeTexts`/`locales` on the pack object anymore.** The former `PersonaTexts` fields — `defaultVoice`, `languageLine`, `fallbackError`, `fallbackNoCredits`, `targetNameFallback`, `helpText`, `infoTitle` — and the `/config` descriptions / group titles / preset descriptions live in the pack's **own `i18n/<lang>.json` folder**, discovered and merged into the engine i18n by the same generate step (`scripts/select-persona.mjs`). `getPersonaTexts(lang)` rebuilds the `PersonaTexts` at runtime from `tRaw(lang, "persona_" + field)` (lang→`en` fallback), falling back to the **engine's own neutral i18n keys** (`NEUTRAL_KEYS`: `fallbackError`→`neutral_fallback_error`, `fallbackNoCredits`→`neutral_fallback_no_credits`, `targetNameFallback`→`neutral_target_name`, `infoTitle`→`neutral_info_title`, resolved via `t()`) so the no-pack build localizes per `cfg.lang` — `defaultVoice`/`languageLine`/`helpText` have no neutral key (empty by design: neutral = no voice/help), plus the live `wakeWords`/`usernameAliases` off the active pack object.

**personaState.** A generic per-chat JSON slot (`chatData.personaState`, D1 column `persona_state`) the persona owns. Each command's `state` slice (`Record<string, PersonaStateField>` = `{type, default, min?, max?}`) contributes to the schema; the engine merges all slices into `getPersonaStateDefaults()` (applied to a new chat and on `/memory forget all`/`… state`) and exposes the `updatePersonaState(ctx, patch)` mutator. The engine stores/merges/defaults it but never interprets the keys — so persona state (a "mood" level, counters, flags) lives entirely in the pack.

Composition is **eager**: the pack is registered by `import "./persona/active"` placed **first** in `index.ts` (before flow/commands/config evaluate), so the `engine-base ∪ registry` maps see it.

**The pack is connected ONLY via the `PERSONA_PACK` env-path — the engine never names a specific pack.** `src/persona/active.ts` is **generated** by `scripts/select-persona.mjs` (run from `prepare`/`pretest`/`precheck`/`predev`): if `PERSONA_PACK` (a path to a pack dir of `*.ts`) is set, it copies them into `src/persona/_pack/` and writes `active.ts` = `import "./_pack"`; otherwise `active.ts` = `import "./default"` (registry `NEUTRAL`: generic voice/help, no fun commands). `active.ts` and `_pack/` are **gitignored**. So this repo carries **zero** persona content — `git grep` clean across code, config and tests.
- **Local dev:** `PERSONA_PACK=<path-to-a-pack> npm test` (or `npm run dev`); the pack's `tests/*.persona.test.mjs` can be copied next to the engine tests (gitignored) to run them too. No `PERSONA_PACK` → neutral engine.
- **Deployment project** (separate repo): pulls this engine, sets `PERSONA_PACK` to its pack, stages the pack's tests, and runs the deploy with its own `wrangler.jsonc` (real worker/resources/secrets).

A pack must implement the `PersonaPack` contract above. The pack reads its **own** env vars (switches, probabilities) via a cast — the engine `Env` type does not name them. Per-call persona readers (`assemblePrompt`/`buildHelp`/`buildInfoStatus`/`messageMentionsBot`/`buildCommandRegex`/`getPersonaTexts`/the `buildPromptLines`/`infoLines`/`adminFlags` hooks) degrade cleanly under `NEUTRAL` (covered by `personaless.test.mjs`).
